Section 10 — D1: Agentic Architecture & Orchestration (27%)

Agents & Workflows

The highest-weight exam domain. Covers the foundational decision between workflows and agents, four core workflow patterns, the agentic loop lifecycle, multi-agent orchestration, and the error handling patterns that separate production-ready systems from fragile ones.

D1 — Architecture 27% D2 — Tool Design 18% D5 — Reliability 15%

Why this section matters most: D1 is 27% of the exam — the single largest domain. The workflows vs. agents decision, the agentic loop's stop_reason check, and multi-agent orchestration patterns are tested across multiple scenarios. The Customer Support Agent (Scenario 1) and Multi-Agent Research System (Scenario 3) both draw heavily from this section.

D1 The Core Decision — Workflows vs. Agents
D1 Workflows vs. Agents — The Fundamental Choice
⚙️ Workflows

A workflow is a predetermined, sequential series of calls to Claude where you know the exact steps in advance. You define the flow; Claude executes each step.

Use when: you can picture the exact steps needed, your app constrains users to specific tasks, and you need predictable, testable behavior.

Advantages: Higher task completion rates. Easier to test (known execution sequence). Claude focuses on one subtask at a time → higher accuracy. Predictable and auditable.

Downside: Less flexible. Requires specific inputs. Requires upfront planning.

workflow = [ describe_image(img), # Step 1 generate_cad_model(desc), # Step 2 render_model(cad), # Step 3 evaluate_rendering(render) # Step 4 → loop if fail ]
🤖 Agents

An agent receives a goal and a set of tools, then dynamically determines which tools to call and in what order. The execution path is not predetermined.

Use when: task details are unclear, user requests are varied and unpredictable, and flexibility is genuinely required over reliability.

Advantages: Handles novel, unpredictable situations. Can combine tools in unexpected ways. Can ask users for clarification mid-task.

Downside: Lower task completion rate. Harder to test and evaluate. Less predictable behavior. Higher cost from open-ended tool use.

💡 Exam signal: The course recommendation is explicit: "Prioritize workflows for reliability. Use agents only when flexibility truly required." When a scenario has a well-defined process, the answer involving a workflow is almost always correct over one involving an agent.

D1 The 4 Core Workflow Patterns
D1 Pattern 1: Chaining — Sequential Steps for Complex Tasks

What it is: Breaking a large task into a series of distinct sequential Claude calls, where each step focuses on one specific subtask and the output of one step feeds into the next.

Why it works: Claude can focus on one aspect of the problem rather than juggling multiple requirements simultaneously. Long prompts with many constraints often fail because the model struggles to honor all constraints at once — chaining solves this by separating concerns into dedicated steps.

When to use: Complex tasks with multiple sequential requirements. When Claude consistently ignores some constraints in a single long prompt. When you need non-LLM processing between steps.

Example — Social media video pipeline:

Step 1: Find trending topics (external tool) Step 2: Claude selects most interesting topic Step 3: Claude researches the topic Step 4: Claude writes the video script Step 5: AI avatar + TTS generates video Step 6: Post to social media
💡 Two-step chaining for constraint violations: Step 1 — generate imperfect output accepting some violations. Step 2 — follow-up prompt asking Claude to fix the specific violations found. This beats trying to pack all constraints into one massive prompt.
D1 Pattern 2: Parallelization — Simultaneous Independent Subtasks

What it is: Breaking a complex task into multiple simultaneous subtasks that run in parallel, then aggregating the results in a final step.

Structure: Input → Multiple parallel Claude calls (each specialized) → Aggregation step → Final output.

Why it works: Each subtask can have specialized prompts, criteria, and tools optimized for that one dimension of the problem. This avoids the "juggling" problem of asking one prompt to consider too many competing factors at once.

Example — Material selection for a part:

Parallel calls: Call A: Evaluate suitability for METAL Call B: Evaluate suitability for POLYMER Call C: Evaluate suitability for CERAMIC Call D: Evaluate suitability for COMPOSITE Aggregation call: Compare all 4 analyses → recommend best material

Benefits: Focused attention per call. Each sub-prompt can be independently optimized and tested. Easy to add new options (just add another parallel call). Faster than sequential for parallelizable work.

💡 Exam signal: The Multi-Agent Research System (Scenario 3) uses this pattern — specialized subagents (web search, document analysis, synthesis) run in parallel and pass results to an aggregating step. Spawning multiple Task tool calls in a single coordinator response is the implementation of parallelization.
D1 Pattern 3: Routing — Categorize, Then Specialize

What it is: A two-step pattern where the first Claude call categorizes the incoming request, and the result of that categorization determines which specialized processing pipeline handles the request next.

Structure: User input → Routing call (Claude categorizes) → One of N specialized pipelines (each with its own prompts/tools).

Key insight: User input only goes to one specialized pipeline, not all of them. Each pipeline is fully optimized for its specific category without compromising the others.

Example — Content generation:

Router: Categorize "Python functions" → "Educational" Route to Educational pipeline: → "Create clear explanations with relatable examples and thought- provoking questions"

When to use: Applications handling diverse request types (customer service, content generation). When different request types require fundamentally different approaches. When a generic prompt can't handle all cases well.

D1 Pattern 4: Evaluator-Optimizer — Produce, Grade, Iterate

What it is: A feedback loop where a Producer generates output and an Evaluator grades it. If the output doesn't meet criteria, the evaluation result goes back to the Producer with feedback for improvement.

Structure:

Input → Producer (generates output) ↓ Evaluator (grades output vs criteria) ↓ Accept? → Output Reject? → Feedback → Producer (loop)

Example — Image to 3D model: Claude describes image (Step 1) → Claude generates CAD model (Step 2) → Render model (Step 3) → Claude compares rendering to original (Step 4 — Evaluator) → If inaccurate, loop back to Step 2 with feedback.

When to use: Tasks with clear quality criteria. Tasks where first-pass output is likely to need refinement. Code generation with test cases (generate → test → fix failures → repeat).

💡 Key distinction: The Evaluator uses objective criteria, not sentiment or self-reported confidence. This connects directly to the escalation triggers anti-pattern in Domain 5 — the same principle applies here.
D1 Programmatic Enforcement & Handoffs
D1 Programmatic Prerequisites for Compliance Workflows
❌ Prompt-Based Enforcement

System prompt: "Never call process_refund unless get_customer has returned a verified ID."

This provides probabilistic compliance. Under normal conditions the model follows it. But under unusual inputs, high load, adversarial prompting, or edge cases, there's a non-zero failure rate.

For financial operations, identity verification gates, and regulatory compliance workflows — a non-zero failure rate is unacceptable.

✅ Programmatic Enforcement

A PostToolUse hook that intercepts the tool call pipeline and blocks process_refund until get_customer has returned a verified customer ID.

The check executes outside the model's control — it cannot be bypassed by model reasoning. This is deterministic compliance.

The rule: When a scenario requires "guaranteed", "always", "must never", or "deterministic" enforcement — the answer is a hook or programmatic gate, not a prompt instruction.

💡 Signal words in exam scenarios: "guaranteed", "deterministic", "financial transactions", "identity verification", "compliance", "regulatory" → hooks/programmatic enforcement. "Guidance", "preferred", "generally" → prompt instructions acceptable.
D1 D5 Escalation Triggers — What Counts & What Doesn't
❌ Invalid Escalation Triggers

Sentiment-based escalation: Triggering when the model detects negative sentiment or frustration. Sentiment is explicitly documented as an anti-pattern — it's an unreliable proxy for actual complexity.

Self-reported confidence: Escalating when the model rates its own confidence below a threshold (e.g., 70%). Self-reported confidence scores are unreliable proxies for actual task complexity.

Attempting resolution before escalating an explicit request: When a customer says "I want to speak to a human," attempting resolution first is a policy violation — not a helpful optimization.

✅ Valid Escalation Triggers

Explicit customer request: Customer asks for a human agent. Escalate immediately with a structured handoff summary. No delay, no "let me try one more thing."

Policy gaps: The scenario involves an exception, edge case, or situation the policy doesn't address (e.g., competitor price matching when policy only covers own-site adjustments).

Inability to make progress: Agent has attempted resolution and cannot advance. Objective inability — not a feeling or confidence score.

Structured handoff summary must include: Customer ID, issue root cause, amount/scope, recommended action, conversation history key points.

💡 Tested in Scenario 1 (Customer Support). Any answer involving sentiment detection or confidence scores as escalation triggers is wrong. The three valid triggers above are the exam-expected answers.
D1 The Agentic Loop — Lifecycle & stop_reason
D1 The Agentic Loop — How Tools Drive Execution

The agentic loop is the core execution pattern for any tool-using Claude application. Understanding every step — and especially stop_reason — is essential for D1.

Loop lifecycle:

while True: response = call_claude(messages, tools) add_assistant_message(messages, response) if response.stop_reason != "tool_use": break # "end_turn" → Claude is done tool_results = run_tools(response) # Execute tools add_user_message(messages, tool_results) # Append results # Loop continues

The two critical stop_reason values:

stop_reasonMeaningAction
"tool_use" Claude wants to call a tool Execute tools, append results, continue loop
"end_turn" Claude has a final answer Break loop, return response to user

Tool result structure (critical details):

{ "type": "tool_result", "tool_use_id": tool_request.id, // MUST match tool_use block ID "content": json.dumps(tool_output), "is_error": False // True if execution failed }
💡 Anti-patterns that the exam tests: (1) Parsing assistant text content to decide whether to continue — wrong, always use stop_reason. (2) Setting an arbitrary iteration cap as the primary termination condition — wrong, stop_reason == "end_turn" is the signal. (3) Assuming the loop always terminates in a fixed number of steps.
D1 Abstract Tools vs. Specialized Tools — Agent Design Principle
❌ Hyper-Specialized Tools

Giving an agent tools like refactor_code, install_dependencies, fix_syntax_error — one function per specific task.

This limits the agent to pre-anticipated scenarios only. The agent can't handle novel combinations or tasks you didn't plan for.

Leads to tool proliferation: if you add every specific case, you'll eventually have too many tools (18+), degrading tool selection reliability.

✅ Abstract, Combinable Tools

Giving an agent generic tools it can combine creatively: bash, read, write, edit, glob, grep.

Claude Code uses this pattern — it has no refactor_code tool. Instead, it uses bash + read + write to accomplish the same result, plus countless other tasks the developers never anticipated.

The combination of simple abstract tools enables the agent to handle situations that weren't explicitly planned, as long as the tools are general enough.

💡 The rule: Give agents tools that can be "pieced together" in unexpected ways. 4–5 flexible tools outperform 18 specialized ones for agent reliability.
D1 Environment Inspection — Read Before You Write

An agent operates blindly without feedback mechanisms. After every action, it needs to inspect the resulting environment state to understand what changed, whether the action succeeded, and what to do next.

Core principle: Agents should always observe before acting on changes. "Read before write" is the file system analog.

Examples across contexts:

ContextInspection Pattern
Computer use (UI)Take screenshot after every click/keystroke to observe the new state
File editingRead file contents before modifying to understand existing structure
Video generationExtract screenshots with FFmpeg at intervals to visually verify output
Code executionCheck test results or compiler output before reporting completion

Benefits: Progress tracking. Error detection. Quality assurance before reporting completion. Adaptive behavior when unexpected results occur.

D1 Multi-Agent Orchestration — Coordinator & Subagents
D1 Hub-and-Spoke Architecture — How Coordinators Work

In multi-agent systems, a coordinator manages all communication, error handling, and routing. Subagents are invoked via the Task tool and operate in isolated contexts.

Coordinator responsibilities:

ResponsibilityWhy it matters
Task decompositionBreaks goal into specialized subtasks with clear scope
Subagent delegationAssigns subtasks to appropriate subagents via Task tool
Result aggregationCombines subagent outputs into coherent final response
Retry logicCoordinator owns all retry decisions — subagents do not self-retry
Error handlingReceives structured errors from subagents and decides recovery path
ObservabilityRoutes all inter-subagent communication through itself for tracking

Spawning parallel subagents: Emit multiple Task tool calls in a single coordinator response. This is the canonical pattern for running subagents in parallel.

// Single coordinator response with 3 parallel Task calls: [ Task(agent="web_search", prompt="...", context=...), Task(agent="doc_analysis", prompt="...", context=...), Task(agent="prior_research", prompt="...", context=...) ]
💡 Critical: To invoke subagents, the coordinator's allowedTools must include "Task". Without this, the coordinator cannot spawn subagents regardless of its system prompt instructions.
D1 Subagent Context Isolation — The #1 Multi-Agent Gotcha
❌ Common Mistake

Assuming a subagent automatically inherits the coordinator's conversation history, context, or tool results.

Delegating a task to a subagent with only the final question, expecting it to "know" what the coordinator already gathered.

Example: Coordinator fetches customer data, then spawns a subagent asking "what's the refund amount?" without passing the customer data — the subagent has no idea what the coordinator found.

✅ Correct Understanding

Subagents start with a completely isolated context — a fresh, clean slate. They receive only what you explicitly pass in their prompt.

All relevant context must be explicitly injected into the subagent's prompt. This includes: prior tool results, intermediate findings, source URLs, document excerpts, structured data — anything the subagent needs to do its job.

For synthesis subagents: pass web search results + document analysis outputs directly in the prompt. Use structured data formats to separate content from metadata.

💡 This is tested frequently. Any answer that says a subagent "inherits" or "has access to" the coordinator's conversation history is wrong. Explicit injection is always required.
D1 Task Decomposition — Scope Risks & Integration Passes

The decomposition risk: Overly narrow task decomposition → incomplete coverage of broad topics. If each subagent handles only a tight slice, the synthesis step may be missing important cross-cutting concerns.

Prompt chaining vs. dynamic decomposition:

ApproachWhen to use
Prompt chaining (fixed sequential pipeline) Predictable multi-aspect reviews. Analyze each file individually → cross-file integration pass.
Dynamic adaptive decomposition Open-ended investigations. Generate subtasks based on discoveries at each step.

The integration pass: Complex multi-agent systems often need an explicit synthesis step after all subagents complete — a coordinator call that combines all results into a coherent, deduplicated output. This step prevents the "sum of parts doesn't equal a whole" failure.

D1 D5 Error Handling in Agentic Systems
D1 Structured Error Semantics — What Subagents Return
❌ Anti-Pattern: Generic Errors

Returning a uniform "Operation failed" string when any tool fails. This hides all context from the coordinator, preventing intelligent recovery.

The coordinator can't distinguish between: a transient timeout (retry), a validation error (fix parameters), a business rule violation (escalate), or a permission error (different auth).

Also anti-pattern: silently returning empty results as success when an operation actually failed. The coordinator proceeds as if data was retrieved, making decisions on nothing.

✅ Canonical Pattern: Structured Error

Subagents return structured error objects that give the coordinator what it needs to decide on recovery:

{ "isError": true, "isRetryable": true, // false for policy violations "errorCategory": "timeout", // transient | validation | // business | permission "description": "Database connection timed out after 30s", "attemptedQuery": "SELECT * FROM orders WHERE ...", "partialResults": [...] // if any were found }

The coordinator then decides:

Error typeisRetryableCoordinator action
Transient (timeout)trueRetry with backoff
Validation (bad input)falseFix params, retry once
Business (policy violation)falseEscalate or skip
PermissionfalseLog, skip, or escalate
D1 D5 Access Failure vs. Valid Empty Result — Different Downstream Handling
❌ They look the same on the surface

Both return an "empty" result set. If a tool always returns [] whether the query failed or genuinely found nothing, the coordinator can't act appropriately.

Treating a timeout as "no results found" means: no retry, no escalation, no error logging — just incorrect reasoning based on missing data presented as a successful empty result.

✅ They require different handling

Access failure (timeout, network error, auth failure): The query couldn't be run. System should retry, escalate, or skip with logging. The data state is unknown.

Valid empty result (successful query, no matching records): The query ran successfully and found nothing. This is information — it tells you the record doesn't exist. No retry needed.

The isError flag + errorCategory in structured error responses is exactly what distinguishes these two cases for downstream coordinator logic.

💡 This distinction is tested. An access failure should trigger retry logic. A valid empty result should not. Conflating the two is a documented anti-pattern in D5.
D1 D1 Anti-Patterns Quick Reference
D1 All D1 Anti-Patterns — The Wrong Answers on D1 Questions
✗
Parsing natural language to determine loop termination. Always check stop_reason == "tool_use", never analyze the assistant's text content to decide whether to continue the loop.
✗
Using an arbitrary iteration cap as the primary loop termination condition. An iteration limit is a safety backstop — not the signal that Claude is done. "end_turn" is the termination signal.
✗
Assuming subagents automatically inherit the coordinator's conversation history. Subagents start with an isolated context. All needed context must be explicitly injected in their prompt.
✗
Subagents self-retrying on failure. In the canonical pattern, subagents return structured errors (isError: true, isRetryable: true/false) and do NOT retry themselves. The coordinator owns all retry logic.
✗
Terminating the entire workflow on a single subagent failure. A partial failure should trigger structured error handling, not full workflow termination. Other subagents may still complete successfully.
✗
Using prompt instructions for deterministic compliance in financial/regulatory workflows. Prompt instructions provide probabilistic compliance. Hooks provide deterministic enforcement. Financial and compliance workflows require the latter.
✗
Overly narrow task decomposition without an integration pass. If each subagent handles only a tight slice, the synthesis step may miss cross-cutting concerns. Include an explicit integration pass for complex decompositions.
✗
Sentiment-based escalation or self-reported confidence as escalation triggers. Both are documented anti-patterns. Valid triggers are: explicit customer request, policy gap, inability to make progress.
✗
Silently returning empty results when an operation actually failed. Empty results as success conceals failures from the coordinator. Always use structured errors with isError: true for actual failures.
✗
Preferring agents over workflows for well-defined processes. The recommendation is to always use workflows where possible. Agents are for genuinely unpredictable tasks where flexibility is truly required.

All The Mirror Trap — The Exam's Most Common Gotcha
❌ The Mirror Trap

The exam uses the correct answer from one question as a distractor in another. For example, "few-shot prompting" is the right answer when the problem is inconsistent output format — but it appears as a tempting wrong answer in a question about a different root cause.

Candidates who memorize answer labels ("few-shot = good", "batch API = async") without understanding the context fall for this repeatedly.

✅ How to Avoid It

Always diagnose the root cause before selecting a solution. Ask: "What is actually broken in this scenario?" Then match the solution to the specific failure mode.

A few-shot example set solves inconsistent output format. It doesn't solve missing tool descriptions, stale context, or non-deterministic compliance. The right answer must address the right problem.

💡 Test this habit: When you reach the answer choices, cover them and state what the root cause is first. Then find the answer that addresses that specific root cause. This single habit catches more mirror traps than any amount of memorization.
Exam Strategy
All How to Read Scenario Questions

CCAF scenarios describe a system with a specific problem or design decision. The question asks you to identify the right approach. Here's a reliable reading strategy:

Step 1 — Identify the constraint. What does the scenario explicitly require? ("guaranteed", "deterministic", "blocking", "real-time", "non-interactive") These words map directly to specific solutions.

Step 2 — Identify the failure mode. What's actually broken? Inconsistent output? Missing context? Unreliable tool selection? Wrong escalation behavior? Each failure mode has a canonical fix.

Step 3 — Apply the right domain lens. Is this an architecture question (D1)? A configuration question (D3)? A prompting question (D4)? The domain often narrows the answer set to 2 candidates.

Step 4 — Eliminate, don't just select. Find the 2 answers that are clearly wrong (wrong tool, wrong scope, wrong timing), then choose between the remaining two based on precision.

💡 Signal words to watch for: "guaranteed" / "deterministic" → hooks. "Blocking" / "pre-merge" → synchronous API. "Overnight" / "weekly" / "latency-tolerant" → batch API. "Shared across team" / "version-controlled" → project-level config. "Personal" / "experimental" → user-level config.

Section 2 — D3: Claude Code Configuration & Workflows (20%)

Skills

Skills are Claude Code's on-demand workflow units — reusable, isolated, and composable. This section covers what skills are, how to create and configure them, how they compare to CLAUDE.md and slash commands, and the anti-patterns that trip up exam candidates.

D3 — Claude Code 20%

Why skills matter on the exam: The three-way distinction between Skills / CLAUDE.md / Slash Commands is directly tested in the Critical Distinctions section and appears as a distractor in D3 scenario questions. D3 What Are Skills?
D3 Skills — Definition & Purpose

A skill is a reusable, on-demand workflow unit in Claude Code. It is defined as a markdown file with YAML frontmatter and lives in the .claude/skills/ directory. When invoked, it runs as an isolated sub-agent — the skill executes its instructions and returns output to the main conversation without polluting the main context.

Core properties:

PropertyValue
Location.claude/skills/ (project) or ~/.claude/skills/ (user)
FormatMarkdown file with YAML frontmatter
InvocationOn-demand — triggered explicitly, not loaded every session
Execution contextIsolated sub-agent (context: fork by default)
OutputReturns result to main conversation; intermediate steps stay isolated
💡 The key contrast: CLAUDE.md is always loaded into every session. Skills are invoked only when needed. This distinction is one of the most frequently tested D3 concepts — if a scenario describes task-specific logic running unnecessarily every session, skills (not CLAUDE.md) are the fix.
D3 The Invocation Model — When Skills Run

Skills are invoked when Claude recognizes a user request that matches the skill's description. The description field in the frontmatter is the model's signal — it must be specific enough that Claude knows when to trigger the skill vs. handle the request inline.

Invocation flow:

User: "Run a security audit on this codebase" ↓ Claude matches request to skill description ↓ Skill spawned as isolated sub-agent (context: fork) ↓ Sub-agent executes skill instructions with its own tool calls ↓ Result returned to main conversation ↓ Main conversation context unchanged (no intermediate noise)

Why isolation matters: Without context: fork, every tool call the skill makes (file reads, bash commands, searches) would appear in the main conversation history, flooding the context and degrading future reasoning quality.

D3 Skill File Structure
D3 Frontmatter Fields — Required & Optional

Every skill file is a markdown document with a YAML frontmatter block at the top, followed by the skill's instruction body.

Anatomy of a skill file:

--- name: security-audit description: Perform a comprehensive security audit of the codebase — check for common vulnerabilities, exposed secrets, dependency issues, and misconfigurations. Use when asked to review security or run a security scan. context: fork --- ## Security Audit Instructions You are performing a security audit. Your task is to: 1. Scan for hardcoded secrets (API keys, passwords, tokens) 2. Check dependency versions against known CVEs 3. Review authentication and authorization patterns 4. Identify SQL injection and XSS risks 5. Summarize findings with severity levels (Critical/High/Medium/Low)

Frontmatter fields:

FieldRequired?Purpose
name✅ YesIdentifier for the skill; matches the filename (kebab-case)
description✅ YesTells Claude when to invoke this skill — must be specific and actionable
contextRecommendedfork (isolated) or inline (current conversation); defaults to fork
💡 Description quality matters: A vague description ("does code stuff") means Claude won't know when to trigger the skill. A good description specifies what the skill does AND when to use it. This connects directly to the tool description quality principle in D2.
D3 Multi-File Skills — Modular Composition

A skill can reference other markdown files to keep complex workflows modular. The main skill file orchestrates the flow and includes sub-files for specific components.

Example — a complex deployment skill:

--- name: deploy-review description: Run pre-deployment checklist including tests, security scan, and changelog generation before a release. context: fork --- ## Pre-Deployment Review Follow these steps in order: @.claude/skills/partials/run-tests.md @.claude/skills/partials/security-check.md @.claude/skills/partials/generate-changelog.md Summarize the results from all three steps.

When to use multi-file skills: When a workflow has distinct phases that can be independently maintained or reused across multiple skills. Avoids duplication — the security-check.md partial can be referenced by both the deployment skill and a standalone security audit skill.

D3 context: fork vs. context: inline
D3 Execution Context — The Isolation Decision
⚙️ context: inline

The skill runs within the current conversation context. Every tool call, intermediate reasoning step, and file read that the skill performs becomes part of the main conversation history.

Result: The main conversation grows with all the skill's intermediate work — file contents, search results, bash outputs — which accumulates in the context window and affects future reasoning.

Use only when: The skill specifically needs to reference or continue from the ongoing conversation (rare).

✅ context: fork (recommended)

The skill spawns as an isolated sub-agent with its own clean context. It executes its instructions independently and returns only the final result to the main conversation.

Result: The main conversation receives only the skill's output — not the hundreds of intermediate tool calls, file reads, and reasoning steps the skill performed to get there. Context stays clean.

Use for: Almost all skills. Security audits, code reviews, test generation, documentation — any workflow that doesn't need to mutate the ongoing conversation.

💡 Default to fork. The exam anti-pattern is using inline for a complex multi-step workflow (security audit, full codebase review) — the context pollution from hundreds of tool calls is the documented failure mode.

D3 D1 Skills vs. Hooks — Model-Driven vs. Deterministic
🎯 Skills — Model-Driven

Skills execute when the model decides the user's request matches the skill's description. Invocation is probabilistic — Claude interprets the request and decides whether to trigger the skill.

Use for: Workflows that enhance Claude's capabilities. The model can adapt, combine skills, or decline if the request doesn't match.

⚙️ Hooks — Deterministic

Hooks execute on specific system events (PreToolUse, PostToolUse, Stop) regardless of model reasoning. They run outside Claude's decision-making — deterministic and cannot be bypassed.

Use for: Compliance rules, security gates, audit logging — anything requiring guaranteed execution regardless of model behavior.

💡 Exam signal: "Guaranteed every time" → hooks. "On-demand workflow" → skills. The word "guaranteed" or "deterministic" in a scenario rules out skills as the answer.
D3 Sharing & Scope
D3 Project-Level vs. User-Level Skills — What Gets Shared
👤 User-Level Skills (~/.claude/skills/)

Stored in the user's home directory. Not version-controlled, not committed to the repo. Only visible to the individual developer on their own machine.

Use for: Personal workflow preferences — e.g., a skill for formatting code in a style you personally prefer, or an experimental workflow you're developing.

New teammates: Will not see or have access to these skills when they clone the repo.

🏢 Project-Level Skills (.claude/skills/)

Stored in the project's .claude/skills/ directory. Committed to version control. All team members get the skills when they pull the repo. CI/CD pipelines can invoke them.

Use for: Team-wide workflows that everyone should be able to invoke — security audits, standardized code reviews, deployment checklists, test generation pipelines.

New teammates: Immediately have access to all project skills after cloning — no manual setup required.

💡 Exam signal: "A new engineer joins the team and should have access to the security audit workflow" → project-level skills. "A developer wants a personal shortcut for their own preferred workflow" → user-level skills. This parallels the CLAUDE.md scope decision.

D1 D4 The Messages API
D1 Request & Response Structure

Minimal request:

import anthropic client = anthropic.Anthropic() # reads ANTHROPIC_API_KEY from env response = client.messages.create( model="claude-3-5-sonnet-20241022", max_tokens=1024, system="You are a helpful assistant.", # optional top-level field messages=[ {"role": "user", "content": "Explain RAG in one paragraph."} ] ) print(response.content[0].text) # the model's reply print(response.stop_reason) # "end_turn" | "max_tokens" | "tool_use" | "stop_sequence" print(response.usage.input_tokens) print(response.usage.output_tokens)

The four stop_reason values — what each means:

stop_reasonMeaningWhat to do
"end_turn" Model finished its response naturally Terminate loop, return response to user
"tool_use" Model wants to call one or more tools Execute tools, append tool_result, continue loop
"max_tokens" Response was cut off at the max_tokens limit Response is incomplete — increase max_tokens or prompt for continuation
"stop_sequence" A custom stop string was encountered Treat as terminal; the stop string itself is not included in the response
💡 D1 critical: "tool_use" means continue the agentic loop. "end_turn" means stop. Never terminate the loop on "tool_use" and never parse the assistant's text to decide whether to continue — always use stop_reason.
D1 Multi-Turn Conversations — Managing the Messages Array

Claude is stateless — it has no memory between API calls. Multi-turn conversation context is maintained by the caller: append each exchange to the messages array and send the full history on every request.

messages = [] # Turn 1 messages.append({"role": "user", "content": "What is RAG?"}) response = client.messages.create(model=MODEL, max_tokens=512, messages=messages) messages.append({"role": "assistant", "content": response.content[0].text}) # Turn 2 — full history included messages.append({"role": "user", "content": "Give me a code example."}) response = client.messages.create(model=MODEL, max_tokens=1024, messages=messages) messages.append({"role": "assistant", "content": response.content[0].text})

Rules: Messages must alternate user → assistant → user. Cannot have two consecutive messages of the same role. The system prompt is separate from the messages array — it is not a message role.

💡 Context growth: Each turn adds tokens to the context window. For long conversations or agentic loops, accumulated tool results can fill the window — trimming verbose tool outputs (D5) is the documented mitigation.
D4 Key API Parameters
D4 Parameter Reference — When to Use Each
ParameterTypePurpose & Exam Notes
model string Which Claude model to use. Named by family + capability tier. Choose based on task complexity and cost constraints.
max_tokens integer Required. Maximum tokens in the response. If hit, stop_reason will be "max_tokens" — response is incomplete.
system string or array System prompt — sets context, persona, and constraints. Injected before the conversation. Not a message role.
messages array Required. Alternating user/assistant turns. Full conversation history must be sent on every request.
temperature float 0–1 0 = deterministic/consistent output. 1 = creative/varied. Use 0 for structured extraction, classification, code. Use higher values for creative tasks.

stream boolean Enable streaming — receive tokens incrementally rather than waiting for the full response.
tools array Tool definitions available to the model. When provided and model chooses a tool, stop_reason becomes "tool_use".
tool_choice object "auto" (default) / "any" (must call a tool) / {"type":"tool","name":"X"} (must call specific tool). See §5 Tool Use.
💡 temperature=0 for extraction: When a scenario involves structured data extraction, classification, or any task requiring consistent repeatable output — set temperature=0. Higher temperatures introduce unwanted variability into deterministic tasks.
D4 Synchronous vs. Batch API — The Critical Decision
D4 Synchronous API vs. Batch API — When to Use Each
⚡ Synchronous API

Blocking — the caller waits for Claude's full response before proceeding. Returns results immediately when complete.

Use when:

  • Pipeline must block on Claude's response before next step
  • Pre-merge CI checks (pipeline waits)
  • Real-time user-facing interactions
  • Multi-turn tool calling within a single session
  • Any workflow where latency matters
# Synchronous — blocks until complete response = client.messages.create( model=MODEL, max_tokens=1024, messages=[{"role": "user", "content": prompt}] ) # Pipeline can now use response.content
📦 Batch API

Asynchronous, non-blocking — submit a batch of requests, receive a batch ID, poll for results later. Up to 50% cost reduction.

Use when:

  • Workload is latency-tolerant (results not needed immediately)
  • Nightly test generation, weekly security audits
  • Large-scale document processing
  • High-volume tasks where cost matters more than speed
# Batch — non-blocking, poll later batch = client.messages.batches.create(requests=[...]) # Come back later: result = client.messages.batches.retrieve(batch.id) # result.processing_status: "in_progress" | "ended"
💡 Hard limit: Batch API does not support multi-turn tool calling within a single request. Each batch request is single-turn. Agentic loops with tool calls require the synchronous API.

✗
Using the Batch API for a blocking pre-merge check. Batch is asynchronous — the CI pipeline cannot wait for it. Pre-merge requires the synchronous API to block until the result arrives.
✗
Using the synchronous API for overnight or weekly batch workloads. Latency-tolerant, high-volume jobs have no reason to block. The Batch API provides 50% cost savings and better throughput for non-blocking work.
✗
Expecting multi-turn tool calling in a single Batch API request. Each batch request is a single-turn operation. Tool call → result → next reasoning requires the synchronous agentic loop.
D4 D5 Prompt Caching
D4 Prompt Caching — Syntax, TTL & When to Use

Prompt caching lets you cache large, stable content blocks (system prompts, tool definitions, document context) so they don't need to be re-processed on every request. Cached tokens are significantly cheaper than uncached input tokens.

Syntax — add cache_control to any content block:

response = client.messages.create( model="claude-3-5-sonnet-20241022", max_tokens=1024, system=[ { "type": "text", "text": "You are an expert code reviewer...\n\n[Large codebase context — 50,000 tokens]", "cache_control": {"type": "ephemeral"} # Mark for caching } ], messages=[{"role": "user", "content": "Review the auth module."}] ) # First request: cache_creation_input_tokens > 0 (paid to create cache) # Subsequent requests: cache_read_input_tokens > 0 (cheaper cached reads) print(response.usage.cache_creation_input_tokens) print(response.usage.cache_read_input_tokens)

Cache properties:

PropertyValue
TTL (time-to-live)5 minutes — cache expires if not accessed within 5 minutes
Cache type"ephemeral" — the only supported type
CostCache creation costs slightly more than normal; cache reads cost significantly less
What can be cachedSystem prompts, tool definitions, large document context, few-shot examples
What cannot be cachedContent that changes every request (the user's current message)

When to use caching: Any scenario where the same large content block is sent repeatedly across many requests — a 50,000-token codebase context, a large system prompt shared across all users, tool definitions sent with every API call.

💡 The 5-minute TTL is testable: If an agentic loop or workflow has gaps longer than 5 minutes between requests, the cache expires and must be re-created. Design for this when prompt caching is part of a long-running workflow.
D4 Streaming
D4 Streaming — When to Stream vs. Wait for Full Response
⏳ Non-Streaming (Default)

Wait for the complete response before processing. Simpler code — one response object with all content and metadata.

Use when: Processing happens after the response is complete (extraction pipelines, batch jobs, tool-use loops where you need the full content to parse tool calls).

response = client.messages.create(...) # response.content is complete # response.stop_reason is available # response.usage is final
⚡ Streaming

Receive tokens incrementally as Claude generates them. Users see text appearing progressively — better perceived responsiveness for long responses.

Use when: Displaying output to a user in real-time. Chat interfaces, long document generation, any user-facing application where waiting for a complete response feels slow.

with client.messages.stream( model=MODEL, max_tokens=2048, messages=[...] ) as stream: for text in stream.text_stream: print(text, end="", flush=True) final = stream.get_final_message()
💡 Streaming does not change billing — you pay for the same input and output tokens regardless of whether you stream. It only affects when you receive the tokens, not how many you pay for.
D4 Model Selection
D4 Model Tiers — Speed, Cost & Capability Tradeoffs

Claude models are organized in three capability tiers. The right choice depends on the complexity of the task, latency requirements, and cost constraints.

TierBest forTradeoff
Haiku (fastest, cheapest) Simple classification, routing decisions, quick extraction from short documents, high-volume tasks where cost matters most Lower reasoning depth. May miss nuance in complex instructions.
Sonnet (balanced) Most production workloads — code generation, analysis, tool use, customer-facing agents, standard agentic workflows Good balance of speed, cost, and capability. The default choice for most scenarios.
Opus (most capable) Complex multi-step reasoning, difficult coding tasks, nuanced analysis, tasks where accuracy is more important than cost or speed Highest cost, slower. Use when other tiers demonstrably fail on the task.

Model ID naming pattern: claude-[family]-[version]-[date] — e.g., claude-3-5-sonnet-20241022. Always pin to a specific model ID in production — using an alias like claude-sonnet-latest means your application behavior can change when Anthropic updates the alias.

💡 Exam pattern: Scenarios describe a task and ask which model is appropriate. High-volume routing logic → Haiku. Production agent with tool use → Sonnet. Architectural analysis requiring deep reasoning → Opus. Match complexity to capability tier.
D4 D3/D4 API Anti-Patterns — Quick Reference
✗
Parsing assistant text content to determine loop termination. Always use stop_reason. Text parsing is explicitly documented as an anti-pattern — model phrasing is not a reliable control signal.
✗
Terminating the loop when stop_reason == "tool_use". "tool_use" means the model wants to call a tool — execute it and continue. Only "end_turn" terminates the loop.
✗
Not checking for stop_reason == "max_tokens". If a response is cut off at max_tokens, the content is incomplete. Treating a truncated response as complete produces downstream errors.
✗
Using an alias model ID (e.g., claude-sonnet-latest) in production. Aliases point to the current model, which changes when Anthropic releases updates. Pin to a specific dated model ID for reproducible production behavior.
✗
Setting high temperature for structured extraction tasks. Temperature adds variability — the opposite of what you want for JSON extraction, classification, or any task requiring consistent, deterministic output. Use temperature=0.
Section 4 — D4: Prompt Engineering & Structured Output (20%)

Prompt Engineering & Evaluation

How to write prompts that produce consistent, reliable outputs — and how to measure whether they're working. Covers core engineering techniques, few-shot examples, chain of thought, evaluation strategies, false positive/negative tradeoffs, and the independent review pattern.

D4 — Prompt Engineering 20% D5 — Context & Reliability 15%

What D4 actually tests: Not whether you know what few-shot prompting is — but whether you can diagnose which technique fixes which problem. Few-shot solves inconsistent output format. Specific categorical criteria solves high false-positive rates. Independent review instances solve generation bias. The exam tests matching the right technique to the right root cause.

D4 Core Prompt Engineering Techniques
D4 Be Clear, Specific, and Direct
❌ Vague Prompt

Prompts that use subjective language leave interpretation to the model:

"Summarize this document. Be thorough but concise. Focus on what's important."

"Thorough but concise" is contradictory. "What's important" is undefined. Different runs produce differently-structured outputs with no way to know if the model followed the intent.

✅ Specific Prompt

Specify format, length, scope, and structure explicitly:

"Summarize this document in 3–5 bullet points. Each bullet should cover one distinct topic. Focus only on decisions made and their rationale — exclude background context and next steps."

Principles: State what you want, not what you don't want. Define output format explicitly. Constrain scope numerically where possible. Remove ambiguity — don't assume the model will interpret vague directives as you intend.

💡 Exam signal: When a scenario shows inconsistent outputs and asks what to fix — if the prompt uses vague language ("be conservative", "be thorough") the answer is to replace vague directives with specific categorical criteria.
D4 XML Tags — Separating Instructions from Content

XML tags help Claude distinguish between the instructions you're giving it and the content it's operating on. Without delimiters, a model may conflate document content with instructions — especially important when the content might contain instruction-like text (prompt injection risk).

You are a contract analyst. Review the contract below and extract key obligations. <contract> {{CONTRACT_TEXT}} </contract> <output_format> Return a JSON array where each item has: - "party": who has the obligation - "obligation": what they must do - "deadline": by when (null if not specified) </output_format>

Benefits of XML tags:

Tag patternPurpose
<document>, <contract>Isolate untrusted content from instructions — reduces prompt injection risk
<instructions>Clearly delineate what the model should do vs. what it's processing
<examples>Wrap few-shot examples to prevent them being treated as instructions
<output_format>Specify response structure without embedding it mid-prompt
<thinking>Used with extended thinking — Claude's reasoning appears here before the answer
💡 Prompt injection defense: When a scenario involves processing untrusted user-supplied or third-party content (web pages, documents, emails), XML tags that delimit the content protect against injection attacks where the content tries to override the system prompt.
D4 Few-Shot Prompting
D4 Few-Shot Examples — The Fix for Inconsistent Output Format
❌ Zero-Shot: Inconsistent Results

Without examples, the model must infer the desired format from instructions alone. Different phrasings, edge cases, and ambiguous instructions produce variable output structures.

# Instruction-only prompt "Classify the customer sentiment as positive, negative, or neutral." # Run 1 output: "Positive" # Run 2 output: "The sentiment is positive." # Run 3 output: "positive (confidence: high)"

Three runs, three different formats. Downstream parsing breaks.

✅ Few-Shot: Consistent Format

Examples demonstrate the exact output format. The model pattern-matches to the examples — far more reliable than describing format in words alone.

Classify customer sentiment. Return only: positive | negative | neutral <examples> Input: "Great product, fast shipping!" Output: positive Input: "Broken on arrival, terrible support." Output: negative Input: "Package arrived on time." Output: neutral </examples> Input: {{CUSTOMER_MESSAGE}} Output:
💡 Root cause match: Few-shot examples fix inconsistent output format. They don't fix missing tool descriptions, stale context, or non-deterministic compliance. The exam uses few-shot as a distractor for problems it doesn't solve.

D4 Chain of Thought Prompting
D4 Chain of Thought — When Reasoning Before Answering Helps

Chain of thought (CoT) prompting instructs the model to reason through a problem step-by-step before producing a final answer. This improves accuracy on tasks that require multi-step reasoning, where jumping to a conclusion often produces errors.

How to trigger CoT:

# Simple trigger "Think through this step by step before answering." # Structured trigger "Before giving your final classification, reason through: 1. What signals in the text support each category? 2. Which signals are strongest? 3. Are there any contradicting signals? Then give your final answer." # XML-delimited thinking "<thinking>Analyze the problem here</thinking> <answer>Give final answer here</answer>"

When CoT helps:

Good fit for CoTPoor fit for CoT
Multi-step math or logic problemsSimple classification with clear categories
Complex code debuggingHigh-volume extraction where latency matters
Nuanced content moderation decisionsStraightforward format conversion
Scenarios with competing factors to weighTasks where the answer is immediately obvious
💡 CoT tradeoff: Adding chain of thought increases output token usage — more reasoning = more tokens = higher cost and latency. For high-volume, low-complexity tasks this cost isn't justified. Use CoT when the accuracy gain outweighs the cost.
D4 Prompt Evaluation
D4 The Evaluation Lifecycle — Build, Measure, Iterate

Prompt evaluation is the systematic process of measuring how well a prompt performs against defined success criteria — before shipping to production and when iterating on prompt changes.

1. Define success criteria → What does "good" look like? Measurable. 2. Build a test suite → Representative inputs + expected outputs → Include edge cases and adversarial examples 3. Choose an evaluation strategy → Programmatic / model-graded / human 4. Establish a baseline → Run current prompt, record metrics 5. Iterate on the prompt → Change one variable at a time 6. Re-evaluate and compare → Did the metric improve? Did anything regress?

Key principle — define criteria before running evals: Success criteria must be defined independently of the model's current outputs. If you write criteria after seeing outputs, you risk fitting the criteria to what the model already does rather than what the product requires.

D4 False Positive / False Negative Tradeoffs
D4 FP / FN Tradeoff — "Be Conservative" Is Not a Criterion
❌ Vague Threshold Instruction

Telling the model to "be conservative" or "err on the side of caution" when classifying content:

"Flag this message if it might contain harmful content. Be conservative — when in doubt, flag it."

Result: High false-positive rate. The model flags ambiguous content it's unsure about — anything that "might" be harmful gets flagged, including genuinely safe content. The downstream team wastes time reviewing false alarms.

"Be conservative" gives the model no basis for distinguishing true positives from edge cases — it just increases overall flag rate.

✅ Specific Categorical Criteria

Define exactly what constitutes a flaggable item. The model has an objective standard to apply:

"Flag this message ONLY if it contains one or more of: - Explicit threats of physical violence toward a named individual - Instructions for illegal weapons modification - Personal financial information (SSN, full card numbers) If the content is aggressive in tone but doesn't meet these criteria: NOT_FLAGGED."

Result: The model can now distinguish clearly between a true positive (explicit threat) and an edge case (rude but not threatening). False-positive rate drops significantly.

💡 Exam signal: A scenario says "the content moderation system flags too much safe content" — this is a high false-positive problem. The fix is specific categorical criteria, not "be less conservative" (that just swings to high false negatives).
D4 Balancing Precision and Recall — The Tradeoff
TermDefinitionHigh rate means
True Positive (TP)Correctly flagged violationGood
False Positive (FP)Safe content incorrectly flagged as violationWasted review time; user friction
False Negative (FN)Actual violation missed / not flaggedPolicy failures slip through
PrecisionTP / (TP + FP) — of everything flagged, how much was actually a violation?High FP → low precision
RecallTP / (TP + FN) — of all actual violations, how many were caught?High FN → low recall

The core tradeoff: Making criteria more strict (to reduce FP) risks increasing FN — some real violations will no longer meet the strict threshold. Making criteria more lenient (to reduce FN) increases FP. The right balance depends on the cost of each error type for the specific use case.

💡 Use case drives the tradeoff: A medical diagnosis system prioritizes recall (catch all real cases, even if it means more false alarms). A content review queue prioritizes precision (only surface real violations to human reviewers, to avoid overwhelming them). Define which error type is more costly before setting criteria.
D4 Independent Review Instances
D4 Self-Review Is an Anti-Pattern — Generation Bias
❌ Self-Review

Asking the same Claude instance that generated code (or any content) to review its own output in the same conversation:

# Same instance, same conversation response = claude.generate_code(task) review = claude.review_code(response) # ❌

Why it fails: The model retains the reasoning context from generation. It "remembers" why it made each decision and will evaluate the output through the same lens that produced it. Errors rooted in a flawed initial assumption will be missed because the reviewer shares that assumption.

This is like asking an author to proofread their own writing immediately after finishing it — they read what they intended to write, not what's actually on the page.

✅ Independent Review Instance

Use a completely separate Claude instance with no knowledge of how the content was generated. The reviewer sees only the output — not the reasoning, context, or assumptions that produced it.

# Two separate API calls, separate instances generation_client = anthropic.Anthropic() review_client = anthropic.Anthropic() code = generation_client.messages.create( messages=[{"role": "user", "content": f"Write code for: {task}"}] ) review = review_client.messages.create( messages=[{"role": "user", "content": f"Review this code for bugs and security issues:\n\n{code.content[0].text}"}] ) # ✅ Fresh context, no generation bias

The reviewer instance will: Challenge assumptions the generator took for granted. Notice missing edge case handling. Flag security issues the generator's framing made invisible.

💡 Tested in Scenario 5 (CI/CD). Any answer that has the same instance generate AND review code is wrong. Two separate API calls = two independent instances = unbiased review. This is also why CI review re-runs must inject prior findings — otherwise the new instance has no knowledge of what was already reviewed.
D4 D4 Anti-Patterns — The Wrong Answers on Prompt Engineering Questions
✗
Using "be conservative" or "err on the side of caution" as a classification threshold. These are vague directives, not criteria. They inflate false-positive rates without giving the model a principled basis for distinguishing true positives from edge cases.
✗
Applying few-shot prompting to fix a problem it doesn't solve. Few-shot fixes inconsistent output format. It doesn't fix missing tool descriptions, stale context, non-deterministic compliance, or wrong tool selection. Match the technique to the root cause.
✗
Using the same Claude instance to generate and then review its own output. Generation bias — the reviewer shares the generator's assumptions and blind spots. Always use a separate instance for review.
✗
Re-running a review instance without injecting prior findings. The fresh instance has no knowledge of what was already reviewed and resolved. It will re-flag already-fixed issues, wasting review cycles.
✗
Assuming a strict JSON schema ensures the extracted values are semantically correct. Schema validation enforces structure (correct types, required fields present). It cannot verify whether an extracted value is accurate — a schema-valid wrong value passes validation.
✗
Defining evaluation success criteria after observing the model's current outputs. Criteria defined post-hoc are fit to what the model already does, not what the product requires. Define criteria independently before running any evals.

D2 Tool Definition & Structure
D2 Tool Anatomy — Name, Description, Input Schema

Each tool is a JSON object passed in the tools array on every API request. The model reads the definitions to decide which tool to call and with what parameters.

The three fields — what each does:

FieldRoleQuality standard
name Identifier the model uses in tool_use blocks Snake_case, descriptive verb-noun: get_customer, process_refund

D2 Tool Description Quality — The Exam's Most-Tested D2 Concept
❌ Poor Descriptions

Too vague — model can't reliably select:

"name": "get_data", "description": "Gets data from the system." "name": "process", "description": "Processes the request."

Ambiguous/overlapping — model misroutes:

"name": "get_customer_info", "description": "Gets customer information." "name": "fetch_user_details", "description": "Fetches details about a user."

These two tools look identical to the model. It will select arbitrarily or inconsistently between them.

✅ High-Quality Descriptions

Specific about what, when, and what NOT to use for:

"name": "get_customer", "description": "Retrieve a verified customer record by email address or customer ID. Use FIRST before any account action. Returns: name, verified status, account tier, and address. Do NOT use for order lookup — use lookup_order instead." "name": "lookup_order", "description": "Fetch order details and current status by order ID (ORD-XXXXXXXX). Use when customer asks about a specific order. Returns status, items, tracking. Do NOT use for customer account info — use get_customer instead."
💡 The "not for" clause is critical when two tools have overlapping domains. Telling the model explicitly what each tool is NOT for prevents the ambiguous overlap that causes misrouting. Without it, the model must guess.

D2 D4 tool_choice — Guarantee Levels
D2 tool_choice: "auto" vs. "any" vs. Forced

D2 Tool Design Principles
D2 Tool Count & Scope — The 4–5 Tool Rule
❌ Too Many Tools (18+)

Giving an agent a large tool set "for maximum flexibility":

tools = [ get_customer, lookup_order, process_refund, escalate_to_human, check_inventory, get_pricing, apply_discount, create_ticket, update_address, cancel_order, send_email, send_sms, get_analytics, run_report, query_database, call_webhook, translate_text, summarize_content # ... 18 tools ]

Result: With 18 similar-looking tools, the model struggles to distinguish between them. Selection reliability degrades. The model may invoke the wrong tool for the task.

✅ Scoped Tool Sets (4–5 tools)

Each agent gets only the tools relevant to its specific role:

# Customer support agent — 4 tools tools = [ get_customer, # Verify identity lookup_order, # Check order status process_refund, # Issue refunds escalate_to_human # Hand off to agent ] # Analytics agent — separate, 4 tools tools = [ query_database, run_report, get_analytics, export_data ]

Result: Each agent has a small, coherent set of tools. Selection is reliable. Out-of-specialization actions are impossible — the refund agent can't accidentally send emails.

💡 The rule: 4–5 well-scoped tools per agent outperform 18 tools for selection reliability. Specialization also prevents out-of-role tool misuse.
D2 Abstract vs. Hyper-Specialized Tools — Combinability
❌ Hyper-Specialized Tools

One tool per anticipated task:

refactor_function() install_dependencies() fix_syntax_error() add_type_hints() format_code() run_linter() generate_docstring()

Problem: Handles only pre-anticipated scenarios. When a novel combination is needed, no tool exists. Adding every case creates tool proliferation — eventually 18+ tools, degrading reliability.

✅ Abstract, Combinable Tools

Generic tools the agent can combine creatively:

bash(command) # Run anything read(path) # Read any file write(path) # Write any file edit(path) # Modify any file

Why it works: Claude Code uses exactly this pattern. With bash + read + write + edit, it can refactor, install dependencies, fix syntax, add type hints, run linters — all without dedicated tools for each. Novel combinations emerge naturally.

The principle: Prefer tools that can be "pieced together" in unexpected ways. Abstract tools handle unanticipated scenarios; specialized tools only handle planned ones.

D2 D5 Structured Error Handling
D2 Structured Errors — isError + isRetryable Pattern
❌ Generic Error Response

Returning the same generic string for every failure type:

def lookup_order(order_id): try: return db.query(order_id) except Exception: return "Operation failed" # ❌

Problem: The coordinator can't distinguish between a timeout (retry-able), a validation error (fix the params), a business rule violation (escalate), or a permission error (different auth path). All failures look identical — intelligent recovery is impossible.

Silent suppression is worse: Returning [] or {} on failure (instead of an error) makes the failure look like a successful empty result. The coordinator proceeds with missing data as if it's complete.

✅ Structured Error Response
def lookup_order(order_id): try: return {"success": True, "data": db.query(order_id)} except TimeoutError: return { "isError": True, "isRetryable": True, "errorCategory": "timeout", "description": "DB timed out after 30s", "attemptedQuery": f"order:{order_id}" } except ValidationError as e: return { "isError": True, "isRetryable": False, "errorCategory": "validation", "description": str(e) }

Coordinator recovery logic based on structured errors:

errorCategoryisRetryableCoordinator action
timeouttrueRetry with backoff
validationfalseFix parameters, retry once
businessfalseEscalate or skip with explanation
permissionfalseLog, skip, or escalate
💡 The is_error flag on tool_result: Set "is_error": True in the tool_result block when execution failed. This signals to Claude that the result represents a failure, not data — prompting it to reason about recovery rather than act on the error payload as if it were valid output.
D2 D5 D2 Anti-Patterns — The Wrong Answers on Tool Design Questions
✗
Vague or minimal tool descriptions. "Gets data." is not a description — it's a placeholder. The description is the model's only signal for when to use the tool. Vague descriptions produce unreliable and inconsistent tool selection.
✗
Ambiguous or overlapping tool descriptions. Two tools that sound the same from their descriptions cause the model to select arbitrarily between them. Add explicit "use this when... not that" language to disambiguate.
✗
Giving an agent 18+ tools. More tools degrade selection reliability. Keep each agent to 4–5 tools scoped to its specialty. Split large tool sets across multiple specialized agents.
✗
Including out-of-specialization tools in an agent's toolkit. A customer support agent with an analytics dashboard tool will occasionally use it inappropriately. Each agent's tools must match its defined role.
✗
Using tool_choice: "auto" when a guaranteed tool call is required. "Auto" permits a conversational text response — stop_reason == "end_turn". Any pipeline that depends on receiving a tool call will break. Use "any" or forced tool_choice.
✗
Returning generic "Operation failed" on tool errors. Hides the error category and retryability from the coordinator. Structured errors with isError + isRetryable + errorCategory enable intelligent recovery decisions.
✗
Returning empty results silently on failure. An empty [] or {} looks like a successful result with no data. The coordinator proceeds on missing data as if it's complete — a documented anti-pattern distinct from returning a structured error.

D5 What is RAG?
D5 RAG — Definition, Purpose & When to Use

Retrieval-Augmented Generation (RAG) is the pattern of retrieving relevant documents from an external knowledge base and injecting them into Claude's context before generating an answer. Claude answers from retrieved content — grounded in actual documents rather than relying solely on training knowledge.

Why RAG solves what prompting alone can't:

ProblemHow RAG addresses it
Context window limitsRetrieve only relevant chunks — not the full knowledge base
Training knowledge cutoffRetrieved documents can be current and continuously updated
Proprietary/internal knowledgeInternal docs, DBs, and codebases Claude was never trained on
Hallucination on specific factsClaude answers from retrieved source material, not inference
💡 Exam signal: When a scenario says Claude doesn't know about internal company policies, recent events, or proprietary data — RAG is the answer. Not fine-tuning (expensive, static), not a larger context window (can't fit everything), not a longer system prompt (same problem).
D5 The Retrieval Pipeline
D5 Offline Indexing & Online Retrieval

Offline (indexing — runs once or periodically):

Documents → Chunk (split into overlapping segments) → Embed (generate vector representations) → Store in vector database (Pinecone, pgvector, etc.)

Online (retrieval — runs per query):

User query → Embed query (same embedding model as indexing) → Semantic similarity search → top-k most relevant chunks → Inject chunks into Claude's context window → Claude generates answer grounded in retrieved content

Chunking strategies:

StrategyBest for
Fixed-size (e.g., 512 tokens with overlap)General purpose; simple to implement
Semantic (sentence or paragraph boundaries)Preserves meaning; better retrieval quality
Hierarchical (summary + detail chunks)Long documents where both overview and specifics are needed
💡 Core principle: Inject only the top-k most relevant chunks — not the entire knowledge base. This is the fundamental design insight of RAG: selectivity is what makes it scale.
D5 D1 Context Window Management
D1 Context Accumulation in Agentic Loops

In a long agentic session, every tool call appends its result to the conversation. Context grows turn by turn:

context = system_prompt # fixed + conversation_history # grows each turn + tool_results # each tool adds its output + pending_response # model's current answer # In a 20-tool-call session: # Each tool result averages 500 tokens → +10,000 tokens # Plus conversation turns → easily 50,000+ tokens total

D5 "Lost in the Middle"
D5 The Lost-in-the-Middle Effect — Position Matters
📉 What Happens

In very long context windows, language models attend more reliably to content at the beginning and end of the context. Information buried in the middle of long inputs is processed less reliably — it may be missed, under-weighted, or misattributed.

Example: A 50-page contract is injected raw. The key liability clause on page 25 (middle of the document) is missed in the analysis. The model correctly surfaces clauses from pages 1–3 and the final pages.

Affects both: Where the question appears in a long prompt, and where the source facts appear in injected documents.

✅ Mitigations

Place critical content at boundaries:

  • Put critical instructions at the beginning of the system prompt
  • Place the most important retrieved chunk first in RAG injection
  • Put the user's key question at the end of a long prompt, not buried after long context

Extract rather than inject raw: For long documents, extract and surface key facts explicitly rather than injecting the full document and hoping the model finds them.

Chunk and select: RAG's top-k retrieval naturally mitigates this — you inject small relevant chunks, not entire documents.

💡 D5 critical. If a scenario describes a long-document pipeline where Claude misses key information, "lost in the middle" is the diagnosis. The fix is restructuring context so critical info is at boundaries — not injecting a longer context window.
D5 Trimming Strategies & Anti-Patterns
D5 Verbose Tool Output — Return 5 Fields, Not 40
❌ Full API Response (40 fields)
{ "customer_id": "C-123", "name": "Jane Doe", "email": "jane@example.com", "phone": "555-0100", "created_at": "2021-03-15T09:22:11Z", "last_login": "2024-11-01T14:07:33Z", "subscription_tier": "premium", "billing_cycle": "annual", "payment_method_type": "credit_card", "card_last_four": "4567", "billing_address": {...}, "shipping_addresses": [...], "preferences": {...}, "notification_settings": {...}, # ... 26 more fields }

Every tool call adds this full object. After 10 tool calls: 10× the tokens consumed in context.

✅ Trimmed Response (5 fields)
{ "customer_id": "C-123", "name": "Jane Doe", "verified": true, "account_tier": "premium", "open_orders": 2 }

Return only what the agent needs for its current task. The tool implementation filters the API response before returning it — the agent never sees the noise.

~90% context reduction per tool call. Over dozens of calls in a long agentic session, this is the difference between a context window that lasts vs. one that fills up mid-task.

💡 D5 anti-pattern: "Verbose tool outputs accumulating in context" is explicitly listed in the exam guide. The fix is trimming at the tool implementation level — not asking the model to ignore fields it's already received.
D5 Summarization — What's Safe vs. What Destroys Precision
❌ NEVER Summarize These

Progressive summarization destroys precision on values where exactness matters downstream:

Original: "$5,420.17" Summarized: "approximately $5,000" ❌ Original: "23.7% conversion rate" Summarized: "about 24%" ❌ Original: "1,247 affected records" Summarized: "about 1,200 records" ❌ Original: "2024-11-15 14:32:07 UTC" Summarized: "recently" ❌

Downstream systems, financial calculations, or Claude's own later reasoning may depend on exact values. Precision loss compounds through a workflow.

✅ Safe to Summarize

Narrative content and resolved context that doesn't carry exact values:

# Old conversation turn (narrative): "Customer explained they ordered the wrong size and would like to exchange..." → OK to summarize: "Customer requested exchange (wrong size)" # Completed background research: [3 paragraphs of context-setting] → OK to summarize into 2 sentences # Resolved subtask result (no numbers): "Authentication step completed successfully — user verified" → Can be dropped entirely once acted upon
💡 Testable rule: If the content contains a number, percentage, date, timestamp, or amount — preserve it exactly. Only summarize narrative and descriptive content where precision isn't at stake.
D5 D5 Anti-Patterns — Context & RAG
✗
Progressive summarization of exact numbers, percentages, or dates. Precision loss is irreversible. Always preserve exact values — "$5,420.17" not "approximately $5,000".
✗
Key findings buried in the middle of long context inputs. Place critical information at the beginning or end of injected context. Information in the middle of very long inputs is less reliably processed.
✗
Verbose 40-field tool outputs accumulating across many turns. Return only the fields the agent needs for its current task. Filter at the tool implementation level.
✗
Injecting entire documents when only specific sections are relevant. Use chunking and top-k retrieval. Inject only relevant chunks — not the full document — to keep context lean and "lost in the middle" risk low.
✗
Resuming a session after prior tool results become stale. If files changed significantly, prior tool results are outdated. Start fresh with an injected structured summary rather than resuming a session that will reason from stale data.

D2 What is MCP?
D2 MCP — Definition, Purpose & Why It Exists

The Model Context Protocol (MCP) is an open standard for connecting AI models to external tools and data sources. It defines a common protocol so that any MCP-compatible AI application can connect to any MCP-compatible server — without custom integration for each pair.

The problem MCP solves:

Before MCPWith MCP
Adding a new tool to an appCustom integration per app per toolBuild one MCP server; any MCP app can use it
Reusing a tool across appsCopy-paste or re-implement for each appConnect multiple MCP clients to the same server
Tool discoveryHardcoded in each applicationServer advertises its capabilities dynamically

Analogy: MCP is to AI tools what USB is to hardware peripherals — a standardized interface that eliminates the need for custom connectors between every device pair.

💡 Key framing: MCP doesn't replace API tool use — it standardizes how tools are exposed and discovered. An MCP tool has the same name, description, input_schema structure as a direct API tool, but it lives on a server and is accessible to any MCP-compatible host.

D2 Transport Types
D2 stdio vs. SSE — Local vs. Remote Transport
📡 SSE (Server-Sent Events)

HTTP-based transport. The MCP server runs as a web service. The client connects over HTTP and receives server messages via Server-Sent Events.

Use for: Remote servers, shared team servers, cloud-deployed MCP services. Multiple clients can connect to the same server simultaneously.

# SSE server config { "mcpServers": { "my-remote-server": { "url": "https://mcp.example.com/sse" } } }
🔌 stdio (Standard I/O)

Process-based transport. The MCP server runs as a child process of the host. Communication happens via stdin/stdout — the host writes requests, the server writes responses.

Use for: Local tools, single-user servers, servers that need filesystem access on the developer's machine. Simpler to develop and debug.

# stdio server config { "mcpServers": { "filesystem": { "command": "npx", "args": ["-y", "@modelcontextprotocol/server-filesystem", "/path/to/allowed/dir"] } } }
💡 Decision rule: Local tool that needs access to local files/processes → stdio. Shared team server or cloud service → SSE.

D3 Claude Code — The Agentic CLI

D3 CI/CD Integration
D3 The -p Flag — Non-Interactive Mode for Automation
❌ Without -p in CI/CD
# In a GitHub Actions workflow: - name: Review PR run: claude "Review this PR for bugs" # ❌ May prompt: "Do you want to proceed?" # No user present → pipeline hangs indefinitely

Without -p, Claude Code may pause to ask for user confirmation, clarification, or approval. In an automated CI/CD environment with no human present, the pipeline hangs until it times out.

✅ With -p (print/non-interactive)
# In a GitHub Actions workflow: - name: Review PR run: | claude -p "Review the changed files in this PR for bugs, security issues, and style violations. Output as JSON." # ✅ Runs to completion, outputs result, # exits without prompting

💡 D3 anti-pattern: "No -p flag in CI/CD invocation" is explicitly listed in the exam guide. Any CI/CD scenario that doesn't use -p has this as a likely root cause of pipeline hangs. The fix is always to add -p.

🔄 Resume (--resume)

Continues a prior session with its full conversation history and tool results intact.

Hooks & SDK

Hooks provide deterministic enforcement that prompt instructions cannot match — they run outside Claude's reasoning and cannot be bypassed. The Agent SDK enables programmatic multi-agent systems. Together they represent the densest exam-signal content per hour in the curriculum (Course 4, ~1 hr).

D1 — Architecture 27% D3 — Claude Code 20%

The PostToolUse enforcement pattern is tested in Scenario 1 (customer support refund gate). The Agent SDK's allowedTools requirement for spawning subagents is tested in Scenario 3 (multi-agent research).

D1 What are Hooks?
D1 Hooks — Deterministic Enforcement Outside the Model
📝 Prompt Instructions — Probabilistic

System prompt: "Never call process_refund unless get_customer has verified the customer's identity first."

The model reads this instruction and usually follows it. Under normal conditions, compliance is very high. But under unusual inputs, adversarial prompting, high cognitive load, or edge cases — there is a non-zero probability of the model skipping the step.

For financial operations, identity verification gates, and regulatory compliance: a non-zero failure rate is unacceptable.

⚙️ Hooks — Deterministic

A PostToolUse hook that intercepts the tool call pipeline and blocks process_refund until get_customer has returned a verified ID.

The check executes as a shell command outside Claude's reasoning. The model cannot reason around it, bypass it, or override it. This is 100% deterministic compliance — not probabilistic.

Hooks are configured in settings.json — not in prompts. They run regardless of what the model is trying to do.

💡 Signal words → hooks: "guaranteed", "always enforced", "deterministic", "financial transaction", "identity verification", "regulatory compliance", "cannot be bypassed". Any of these in a scenario → hook is the answer, not a prompt instruction.
D1 Hook Types & Events
D1 The Four Hook Types — When Each Fires
Hook typeWhen it firesCan block?ReceivesUse for
PreToolUse Before Claude's tool call is executed ✅ Yes — non-0 exit blocks the call Tool name + input parameters Validation gates, pre-authorization checks, blocking disallowed operations
PostToolUse After a tool call completes and returns its result ❌ No (result already happened) Tool name + input + output Audit logging, state tracking, triggering downstream actions based on what a tool returned

PreCompact Before Claude compacts/summarizes a long conversation ❌ No Conversation to be compacted Saving important context before compaction, audit trails
💡 PreToolUse vs. PostToolUse: PreToolUse can PREVENT a tool from running (blocking gate). PostToolUse runs AFTER the tool has already executed — it can't undo the action, but can log, trigger, or track state based on what happened. For compliance gates (block refund until verified), use PreToolUse or a PostToolUse that sets state to be checked by a later PreToolUse.
D3 Hook Configuration
D3 Configuring Hooks in settings.json

Hooks are defined in settings.json under the hooks key. Each hook type maps to an array of matchers, each specifying which tool to match and which command to run.

# .claude/settings.json { "hooks": { "PreToolUse": [ { "matcher": "process_refund", "hooks": [{ "type": "command", "command": "check-customer-verified.sh" # Exits 0 = allow; exits non-0 = block tool call }] } ], "PostToolUse": [ { "matcher": "Bash", # Matches all Bash calls "hooks": [{ "type": "command", "command": "audit-log-bash-call.sh" # Logs all bash commands for security audit }] } ], "Stop": [ { "matcher": "*", # Matches all Stop events "hooks": [{ "type": "command", "command": "notify-session-complete.sh" }] } ] } }

matcher: The tool name to match. Use "*" to match all tools of that type. Exact tool name to match a specific tool.

Exit codes: For PreToolUse hooks, exit 0 = allow the tool call to proceed; exit non-0 = block the tool call. Claude receives a message that the hook blocked the action.

D1 Programmatic Enforcement Pattern
D1 PostToolUse Compliance Gate — The Canonical Pattern

The Scenario 1 (Customer Support) pattern: prevent process_refund from being called until get_customer has successfully verified a customer identity.

# .claude/settings.json { "hooks": { "PostToolUse": [ { "matcher": "get_customer", "hooks": [{ "type": "command", "command": "mark-customer-verified.sh" # Sets: echo $CUSTOMER_ID > /tmp/verified_customer }] } ], "PreToolUse": [ { "matcher": "process_refund", "hooks": [{ "type": "command", "command": "require-customer-verified.sh" # Checks: if [ ! -f /tmp/verified_customer ]; exit 1; fi # Exit 1 BLOCKS process_refund if no verified customer }] } ] } }

Why this is deterministic: The require-customer-verified.sh script runs as an OS process. It cannot be overridden by model reasoning. If the file doesn't exist, exit code 1 blocks the tool call — period. No amount of adversarial prompting can make the model bypass an OS-level check.

Compare to prompt instruction: "Never call process_refund without a verified customer" can be forgotten, reasoned around, or overridden by a long conversation that shifts the model's context. The hook cannot.

D1 D3 Hooks & SDK Anti-Patterns
✗
Using a prompt instruction for deterministic compliance in financial or regulatory workflows. Prompt instructions are probabilistic — the model can occasionally miss or rationalize around them. For guaranteed enforcement, use a PreToolUse hook with an exit code that blocks the tool call.
✗

✗

✗
Assuming subagents automatically inherit the coordinator's context. Subagents start with completely isolated context. The coordinator must explicitly inject any needed findings, constraints, or data in the Task tool invocation prompt.
✗

D1 Hooks vs. Prompts — Deterministic vs. Probabilistic Compliance

D3 Plan Mode vs. Direct Execution — When to Explore First
❌ Common Mistake

Using direct execution mode for a task that spans multiple files, involves architectural decisions, or has several valid implementation approaches. Or inversely — engaging plan mode for a single-file bug fix with a clear stack trace.

✅ Correct Understanding

Plan mode is for: complex multi-file changes, architectural decisions with multiple valid approaches, large-scale refactors, or any task where committing to a direction prematurely creates costly rework.

Direct execution is for: single-file changes, well-scoped bugs with a clear stack trace, adding one validation check, or any task where scope is fully understood upfront.

You can combine them: plan mode for investigation, direct execution for implementation once the path is clear.

💡 Exam signal: "Adding a single validation function to one file" → direct execution. "Refactoring authentication across 12 service files" → plan mode. Watch for scale and certainty signals.
D3 Session Resumption vs. Fresh Start — Stale vs. Valid Context
❌ Common Mistake

Resuming a prior session after significant file changes and expecting Claude to have accurate context. Or starting a fresh session when prior tool results are still valid and the work was simply paused.

✅ Correct Understanding

Use --resume <session-name> when prior context is still valid and you're continuing interrupted work on the same codebase state.

Start fresh with an injected structured summary when prior tool results are stale (files have changed significantly). A stale resumed session will confidently reason from outdated information.

When resuming, tell the session specifically which files changed for targeted re-analysis — don't trigger full re-exploration.

💡 Exam signal: If a scenario mentions "significant file changes since the last session" → fresh start + injected summary. If changes are minor or session was just interrupted → resume.
D2 Tool Behavior & Selection
D1 stop_reason: "tool_use" vs. "end_turn" — Loop or Terminate?
❌ Common Mistake

Terminating the agentic loop when stop_reason is "tool_use", or continuing the loop by parsing the assistant's text content to decide whether to keep going.

Parsing natural language for loop termination is an explicit anti-pattern — text output is not a reliable control signal.

✅ Correct Understanding

stop_reason: "tool_use" → the model wants to call a tool. Execute the tool, append the tool_result to conversation history, and send the next iteration.

stop_reason: "end_turn" → the model has finished. Terminate the loop.

The loop lifecycle is: send request → inspect stop_reason → if "tool_use": execute tool, append result, iterate → if "end_turn": stop.

💡 Exam signal: Any question about agentic loop termination — the correct control signal is always stop_reason, never text parsing or arbitrary iteration limits.
D2 tool_choice: "auto" vs. "any" vs. Forced — Guarantee Levels
❌ Common Mistake

Using tool_choice: "auto" in a workflow that requires a guaranteed tool call (e.g., structured data extraction pipeline). The model may instead return a conversational text response, breaking the downstream schema expectations.

✅ Correct Understanding

"auto" — Model decides: may call a tool OR return plain text. Default behavior.

"any" — Model must call a tool (any tool from the available set). Cannot return plain text. Use when you need guaranteed tool invocation but are flexible about which tool.

forced: {"type": "tool", "name": "extract_invoice"} — Model must call this specific named tool. Use for guaranteed schema-compliant JSON output in extraction pipelines.

💡 Exam signal: "Guaranteed structured output" → forced tool. "Must call a tool, any tool" → "any". "Let the model decide" → "auto". Don't confuse "any" with "auto" — "any" is a guarantee.
D2 D5 Access Failure vs. Valid Empty Result — Different Recovery Paths
❌ Common Mistake

Treating both an access failure (timeout, permission error) and a valid empty result (query succeeded, no matching records) as the same outcome. Using a generic "Operation failed" error response that hides which type occurred.

✅ Correct Understanding

Access failure (timeout, auth error) → isError: true, isRetryable: true → coordinator should retry or try alternative approach.

Valid empty result (query succeeded, zero matches) → isError: false, return empty array → coordinator should treat as a legitimate "no results" state and move forward.

The surface appearance is identical (no data returned), but downstream handling is completely different. Structured error metadata enables the coordinator to make the right recovery decision.

💡 Exam signal: "How should a subagent communicate that a database query returned no results vs. timed out?" — Always use structured error semantics, not plain text or generic status codes.
D3 Configuration Scope & Hierarchy
D3 Project-Level vs. User-Level Config — What Gets Shared
❌ Common Mistake

Placing team-wide standards (coding conventions, testing requirements, deployment rules) in ~/.claude/CLAUDE.md (user-level). Teammates won't see these settings — they only apply to your own Claude sessions.

✅ Correct Understanding

Project-level (.claude/CLAUDE.md or root CLAUDE.md): Committed to version control. All team members and CI/CD pipelines pick these up. Use for: team coding standards, project conventions, testing requirements.

User-level (~/.claude/CLAUDE.md): Your personal preferences only. Not version-controlled, not shared. Use for: personal workflow habits, output format preferences, personal tool shortcuts.

💡 Exam signal: "A new engineer joins the team and should automatically follow the same coding standards" → project-level. "A developer wants personal shortcuts for their own sessions" → user-level.
D3 Skills vs. CLAUDE.md vs. Slash Commands — Invocation Model
❌ Common Mistake

Putting task-specific workflow logic (e.g., "run security audit") directly in CLAUDE.md so it runs every session — polluting context with instructions that are rarely relevant. Or manually re-typing common workflows each time instead of using skills.

✅ Correct Understanding

CLAUDE.md → Always-loaded universal standards. Team-wide context that should be present in every interaction (project structure, non-negotiable conventions).

Skills (in .claude/skills/) → On-demand invocation for task-specific workflows. Runs in isolated sub-agent context (context: fork) so output doesn't pollute main conversation. Triggered explicitly.

Slash commands (.claude/commands/) → Explicit invocation of parameterized reusable commands. Version-controlled and shared.

💡 Mnemonic: Always-on / On-demand / Explicit-call = CLAUDE.md / Skills / Slash commands.
D1 Agent Architecture
D1 Subagent Context Isolation — What Subagents Inherit
❌ Common Mistake

Assuming a spawned subagent automatically has access to the coordinator's conversation history — prior tool results, gathered context, constraints established in earlier turns. Writing coordinator prompts that reference "what we discussed" without explicitly passing that context.

✅ Correct Understanding

Subagents have fully isolated context. They start with only what is explicitly included in their invocation prompt via the Task tool.

If the coordinator gathered research results, customer data, or constraints in earlier turns, those must be explicitly passed in the subagent's prompt. Nothing is inherited automatically.

This is intentional — it enables subagents to be scoped to specific subtasks without reasoning over irrelevant prior context.

💡 Exam signal: If a scenario says a subagent "doesn't have access to earlier findings" and asks why — the answer is always context isolation, not a bug. The fix is explicit context passing in the Task tool invocation.
D1 D2 Coordinator Owns Retry Logic — Subagents Don't Self-Retry
❌ Common Mistake

Building retry logic directly into subagents — having them silently retry failed tool calls, swallow errors, or return empty results on failure as if the operation succeeded. This hides error context from the coordinator.

✅ Correct Understanding

In canonical multi-agent architecture, subagents return structured errors to the coordinator using the isError: true, isRetryable: true/false pattern. They do not self-retry.

The coordinator decides: retry the subagent, try an alternative approach, escalate, or terminate the workflow based on the structured error metadata.

This design preserves observability and gives the coordinator full situational awareness to make intelligent recovery decisions.

💡 Exam signal: "Where should retry logic live in a multi-agent system?" → always the coordinator. A subagent that self-retries or silently suppresses errors is an anti-pattern.
D5 Escalation Triggers — What Actually Warrants Escalation
❌ Common Mistake

Escalating based on detected negative sentiment ("customer sounds frustrated"), self-reported confidence scores ("I'm only 60% confident"), or proactively attempting resolution before escalating when a customer explicitly asked for a human.

✅ Correct Understanding

The three legitimate escalation triggers are:

1. Customer explicitly requests a human agent — escalate immediately, do not attempt resolution first.

2. Policy gap or exception — the situation falls outside defined policy boundaries (e.g., competitor price-matching when policy only covers own-site adjustments).

3. Inability to make meaningful progress — multiple resolution attempts have failed and the issue isn't resolvable within agent capabilities.

Sentiment and confidence scores are explicitly documented anti-patterns as escalation signals.

💡 Exam signal: Any answer that escalates based on sentiment or confidence scores is wrong. Watch for "customer expressed frustration" as a distractor — it's never a sufficient trigger on its own.
D4 API & Workflow Selection
D4 Batch API vs. Synchronous API — Blocking vs. Non-Blocking
❌ Common Mistake

Using the Batch API for pre-merge code review checks (blocking workflow), or using the synchronous API for overnight nightly test generation across thousands of files (latency-tolerant, non-blocking).

✅ Correct Understanding

Synchronous API: blocking workflows where the process waits for Claude's response before continuing. Use for: pre-merge checks, real-time user interactions, any pipeline where the next step depends on this result.

Batch API: non-blocking, latency-tolerant workloads. Use for: overnight reports, weekly audits, nightly test generation, batch document processing. Note: does not support multi-turn tool calling within a single request.

Use custom_id fields to correlate batch request/response pairs when processing results later.

💡 Exam signal: "Pre-merge CI check" → always synchronous. "Weekly batch analysis" → always batch. The words "blocking," "pipeline must wait," and "real-time" point to synchronous.
D1 D4 Programmatic Enforcement vs. Prompt Guidance — Compliance Guarantees
❌ Common Mistake

Relying on prompt instructions to enforce compliance rules in financial, identity verification, or regulatory contexts. Assuming that a well-written system prompt can guarantee a specific sequence of operations without fail.

✅ Correct Understanding

Prompt instructions have a non-zero failure rate — they are probabilistic guidance, not guarantees. For workflows where the cost of a compliance failure is high (financial transactions, identity verification, regulatory requirements), prompt instructions are insufficient.

Programmatic enforcement via hooks, prerequisite gates, or orchestration logic provides deterministic guarantees. The compliance check cannot be reasoned around or skipped.

💡 Exam signal: Any scenario mentioning "regulatory," "financial transaction," "identity verification," or "must always verify before proceeding" → programmatic enforcement is the answer, not improved prompt wording.
D4 JSON Schema — What It Prevents vs. What It Doesn't
❌ Common Mistake

Assuming that using a strict JSON schema in a tool call guarantees both structural validity and business-logic correctness — e.g., that line items will always sum to the stated total, or that a date range will always be logically valid.

✅ Correct Understanding

Strict JSON schemas eliminate syntax errors (malformed JSON, missing required fields, wrong types). This is a genuine and valuable guarantee.

Schemas do not prevent semantic errors — the output can be perfectly valid JSON while containing logically inconsistent values (line items not summing to the stated total, a start date after an end date, fabricated values for missing fields).

Design schema fields as optional/nullable when the source document may not contain the information — prevents the model from fabricating values to satisfy required fields.

💡 Exam signal: "Schema-compliant output" → JSON is valid. "Semantically correct output" → schema alone doesn't guarantee this. Separate concern requiring post-processing validation.
D2 D5 Error Semantics
D2 Structured Error Response vs. Generic "Operation Failed"
❌ Common Mistake

Returning a uniform "Operation failed" or empty result from an MCP tool regardless of why it failed. The coordinator receives no signal about whether to retry, try an alternative, escalate, or abort.

Silently converting a failure into a success response (returning [] on timeout) is the most damaging variant — the coordinator proceeds as if the step succeeded.

✅ Correct Understanding

MCP tools should return structured error metadata that enables intelligent coordinator recovery:

{ isError: true, errorCategory: "transient", // transient | validation isRetryable: true, // | business | permission description: "DB timeout after 5000ms" }

Error categories and their retry semantics:

• Transient (timeouts, network) → isRetryable: true

• Validation (bad input format) → isRetryable: false (fix input first)

• Business rule (policy violation) → isRetryable: false

• Permission (auth failure) → isRetryable: false

💡 Exam signal: "The coordinator can't tell whether to retry or give up" → the root cause is always missing structured error metadata, not a coordinator logic bug.
D4 Review & Self-Evaluation
D4 Self-Review vs. Independent Review Instance
❌ Common Mistake

Using the same Claude instance that generated code to also review it. The model retains the reasoning context from generation — it's less likely to question choices it just made and will tend to confirm rather than challenge its own decisions.

✅ Correct Understanding

Use an independent review instance — a fresh Claude invocation that has not seen the generation reasoning. Without that context, it reviews the code on its merits rather than through the lens of "this is what I intended."

This applies directly to CI/CD scenarios: the Claude that generated a PR should not be the same instance reviewing it. Spin up a separate instance for the review step.

When re-running a review (e.g., after a fix), inject prior review findings so the new instance doesn't duplicate already-resolved comments.

💡 Exam signal: "Improve code review accuracy in CI" → independent instance, not re-prompting the same one. The limitation is architectural, not prompt-engineering.
D3 Grep vs. Glob — Content Search vs. Path Matching
❌ Common Mistake

Using Grep to find files by name/extension pattern, or using Glob to search inside file contents for a function name or import statement. They solve different problems.

✅ Correct Understanding

Grep — searches inside file contents for patterns. Use when you need: function names, error messages, import statements, any text that appears within files.

Grep "useAuthToken" src/ → finds every file containing that string.

Glob — matches file paths by pattern. Use when you need: all files of a type, files in a specific directory, files matching a naming convention.

Glob "**/*.test.tsx" → returns all test files regardless of location.

Typical workflow: Glob to find candidate files by path → Grep to find specific content within those files → Read to load and understand specific files.

💡 Mnemonic: Grep = Grab content inside. Glob = Glob file paths together.

Anti-Patterns Library

D1 Agentic Loop Control
✗
Parsing natural language to determine loop termination. Checking the assistant's text output for phrases like "I'm done" or "task complete" as the signal to stop the agentic loop. Text content is probabilistic and unreliable. The only correct control signal is stop_reason: "end_turn".
✗
Using arbitrary iteration caps as the primary loop stop condition. Terminating after a fixed number of turns (e.g., "stop after 10 iterations") without inspecting stop_reason. This causes premature termination on complex tasks and non-termination if the model gets stuck. Caps can exist as safety rails but never as the primary exit condition.
✗
Checking assistant text content as a completion indicator. Variant of the above: scanning assistant message text for completion signals rather than using the structured stop_reason field. Model-driven decision-making via structured signals, not text heuristics.
D2 Tool Design & Distribution
✗
Minimal tool descriptions → unreliable tool selection. Defining tools with short, vague descriptions like "Analyzes content" without specifying input formats, example use cases, edge cases, or boundary conditions. Tool descriptions are the primary mechanism the model uses for tool selection — minimal descriptions cause random or incorrect tool routing.
✗
Ambiguous / overlapping tool descriptions → misrouting. Two tools with nearly identical descriptions (e.g., analyze_content and analyze_document described in the same terms) cause the model to pick the wrong one inconsistently. Each tool needs a clearly differentiated description with defined scope boundaries.
✗
Too many tools per agent (18 instead of 4–5) → degraded selection reliability. Giving an agent a large, undifferentiated toolset forces it to reason across too many options simultaneously. Tool selection reliability degrades measurably as toolset size grows. Scope tool access: each agent should only see the 4–5 tools relevant to its role.
✗
Agents with out-of-specialization tools → misuse. Giving a document analysis subagent access to process_refund or other unrelated tools. The agent will occasionally invoke them incorrectly. Scoped tool access prevents this class of error.
✗
Using tool_choice: "auto" when a guaranteed tool call is required. In structured output extraction pipelines, "auto" allows the model to return plain text instead of calling the extraction tool. Use "any" (any tool, guaranteed) or forced tool selection (specific tool, guaranteed) when the downstream pipeline requires schema-compliant output.
✗
Uniform "Operation failed" error responses. Returning the same generic error for all failure types (timeouts, permission errors, business rule violations, validation errors). The coordinator cannot determine whether to retry, change approach, escalate, or abort. Always return structured errors with isError, isRetryable, errorCategory, and a human-readable description.
✗
Silently suppressing errors by returning empty results as success. Returning { results: [] } (success-shaped response) when a tool actually failed. The coordinator treats the step as complete and proceeds on false premises. An access failure must return isError: true, not an empty success response.
D1 Multi-Agent Orchestration
✗
Overly narrow task decomposition → incomplete topic coverage. Decomposing "analyze this codebase for security issues" into individual file-by-file subtasks without a cross-file integration pass. Narrow decomposition misses issues that span multiple files or require reasoning across components. Always include an integration / synthesis step.
✗
Subagent self-retry (instead of returning structured errors to coordinator). Building retry loops inside subagents so they silently attempt recovery. This hides error context from the coordinator, prevents intelligent recovery decisions, and undermines the hub-and-spoke observability model. Retry logic belongs in the coordinator.
✗
Terminating the entire workflow on a single subagent failure. When one subagent times out or fails, aborting the full multi-agent workflow. The coordinator should handle partial failures gracefully: retry the failed step, try an alternative, use partial results, or escalate — not terminate everything.
✗
Assuming subagents inherit coordinator context automatically. Writing coordinator logic that spawns subagents via the Task tool without explicitly passing needed context (prior tool results, constraints, gathered data). Subagents have fully isolated context — they only know what's in their invocation prompt.
✗
Using prompt instructions alone to enforce deterministic compliance. Relying on system prompt wording to guarantee that financial operations, identity verification steps, or regulatory checks are always performed in sequence. Prompts have a non-zero failure rate. Financial and compliance workflows require programmatic enforcement via hooks or prerequisite gates.
D5 Context Management & Reliability
✗
Progressive summarization of numerical values, dates, and percentages. Condensing tool results that contain exact figures (order totals, refund amounts, dates, percentages) into narrative summaries ("the customer had a recent large order"). These specifics are lost and cannot be recovered. Extract transactional facts into a persistent "case facts" block instead.
✗
"Lost in the middle" — placing key findings in the center of long inputs. Models reliably process content at the beginning and end of long context windows. Findings buried in the middle may be ignored. Place key summaries and critical findings at the beginning of aggregated inputs, use explicit section headers.
✗
Verbose tool outputs accumulating in context unchecked. Appending full tool responses (e.g., 40-field customer records when only 5 fields are relevant) to conversation history without trimming. Token consumption grows with each turn, eventually crowding out useful context. Trim tool outputs to relevant fields before appending.
✗
Resuming a session when prior tool results are stale. Using --resume after significant file changes and allowing the session to reason from outdated tool results as if they're current. Start fresh with an injected structured summary when the codebase or data state has materially changed since the prior session.
✗
Heuristic customer selection when multiple matches are found. When a customer lookup returns multiple records, automatically selecting the "most likely" one based on proximity or recency. The correct response is to request additional identifying information from the customer — never guess.
D4 Prompt Engineering & Output
✗
Vague instructions like "be conservative" instead of explicit categorical criteria. Instructions such as "flag potential issues conservatively" do not reduce false positives — the model has no actionable definition of "conservative." Replace with specific categorical criteria: "flag only when the comment directly contradicts the observed code behavior."
✗
Assuming strict JSON schemas prevent semantic errors. A schema-compliant JSON response can still contain logically inconsistent values (line items not summing to total, start date after end date, fabricated values for fields that didn't exist in the source). Schema validates structure, not business logic. Add post-processing validation for semantic correctness.
✗
Required fields in JSON schemas when source data may be absent. Marking all extraction fields as required forces the model to fabricate values for information not present in the source document. Design schema fields as optional/nullable when the source document may not contain the information.
✗
Self-review: using the same instance that generated output to review it. The generating instance retains its reasoning context and will tend to confirm rather than challenge its own decisions. Use an independent review instance without prior generation context for more effective quality assessment.
✗
Using detailed instructions alone when few-shot examples would work better. For tasks requiring consistently formatted output or correct handling of ambiguous cases, detailed instructions often produce inconsistent results. 2–4 targeted few-shot examples demonstrating the expected input→output transformation are more effective than additional instruction text.
D3 Claude Code Configuration & Workflows
✗
Putting task-specific workflow logic in CLAUDE.md. Placing rarely-used procedures (e.g., "run the full security audit workflow") in CLAUDE.md so they load in every session. This pollutes context with instructions that are irrelevant most of the time. Task-specific workflows belong in Skills (.claude/skills/) or slash commands, invoked on demand.

Shared standards, coding conventions, and project requirements must be in project-level config (root CLAUDE.md or .claude/CLAUDE.md) so they're committed to the repo.

The overhead is unnecessary — direct execution is appropriate for single-file, clearly scoped changes where the solution is known.
✗
Running without -p flag in CI/CD pipelines. Invoking Claude Code in interactive mode in an automated pipeline. Use the -p (or --print) flag for non-interactive execution in CI contexts. Combine with --output-format json and --json-schema for structured output that downstream pipeline steps can parse.
✗
Re-running a review instance without injecting prior findings. When re-running a CI code review after a fix, starting with a fresh instance that has no knowledge of previous review comments. The instance will re-flag already-resolved issues. Always inject prior review findings so the new instance can focus on what changed.
D4 API & Integration Selection
✗
Using the Batch API for blocking pre-merge checks. The Batch API is asynchronous and non-blocking — it cannot be used in a pipeline that must wait for Claude's response before proceeding. Pre-merge CI checks, real-time user interactions, and any blocking workflow require the synchronous API.
✗

✗

Confidence self-reporting is an unreliable proxy for actual task complexity. Use objective triggers: policy gaps, explicit customer requests, inability to progress.
✗
Attempting resolution before escalating when a customer explicitly demands a human. When a customer says "I want to speak to a human," attempting to resolve the issue first before escalating. Integration pass: After all specialized subagents complete, a final coordinator call combines, deduplicates, and structures the results. Without this step, output is a pile of independent reports — not a coherent synthesis.

Likely trap answers:

✗
Assuming the synthesis subagent automatically receives all prior search results. This is the most common D1 trap. Context isolation is absolute — nothing is inherited. "Summarize our research" fails if results weren't explicitly injected.
✗
Giving each subagent access to all 18 research tools. Each subagent should have only the 4–5 tools relevant to its specialty. Tool proliferation degrades selection reliability.
✗
Terminating the entire workflow when one subagent fails. If the prior research subagent times out, web search and document analysis can still complete. Use structured error handling, not full termination.
✗
Having subagents retry their own tool failures. Subagents return structured errors (isError: true, isRetryable: true/false). The coordinator decides whether to retry — subagent self-retry hides context from the coordinator.
💡 The synthesis subagent question: Any answer that says the synthesis agent "has access to" or "sees" earlier subagent results without explicit injection is wrong. Explicit context passing in the Task invocation is always required.

D4 D5 Scenario 6 — Structured Data Extraction

System: An extraction pipeline that reads documents (invoices, contracts, reports) and outputs structured JSON for downstream integration. Questions probe schema design, tool_choice, context management, and edge case handling.

What it tests:

Key concepts to nail:

tool_choice for guaranteed extraction: Use forced tool ({"type": "tool", "name": "extract_invoice"}) when the downstream pipeline requires schema-compliant JSON. "auto" allows the model to return a conversational text response instead — breaking the pipeline. "any" guarantees a tool call but allows tool selection flexibility.

Schema limitations: A strict JSON schema guarantees structural correctness — right types, required fields present. It does NOT guarantee that extracted values are accurate. A schema-valid "invoice_total": 0.00 is structurally correct but semantically wrong if the actual total is $5,420. Semantic correctness requires additional validation.

Optional fields: If a source document might not contain a field (e.g., a discount that only exists on some invoices), mark it optional with a null default — not required. Required fields on absent data cause hard extraction failures.

"Lost in the middle": Claude more reliably extracts data at the beginning or end of long documents. Critical fields buried in the middle of a 50-page contract are more likely to be missed or mis-extracted.

Likely trap answers:

✗
Using tool_choice: "auto" in a pipeline that requires structured JSON output. "auto" allows a text response. For guaranteed schema-compliant output, force the specific extraction tool. This is the most direct D4 trap in this scenario.
✗
Claiming a strict JSON schema ensures extracted values are correct. Schema validates structure, not semantics. A correctly-typed wrong value passes schema validation. Human review or a separate validation step is needed for semantic correctness.
✗
Making all fields required in the extraction schema. Source documents vary — some invoices don't have discount fields, some contracts don't have expiry dates. Required fields on optional source data cause hard failures. Use optional with null defaults.
✗
Progressively summarizing extracted numbers, percentages, and dates. Summarization loses precision. "Approximately $5,000" is not equivalent to "$5,420.17" for financial downstream systems. Preserve exact values through the pipeline.
💡 The schema trap: Any answer claiming "the JSON schema ensures correct extraction" or "schema validation is sufficient" is wrong. Schemas prevent structural errors only. Semantic correctness requires additional validation.
